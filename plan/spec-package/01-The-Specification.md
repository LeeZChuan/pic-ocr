# 核心规格文档（The Specification）

## 1. 产品目标
提供一个“上传图片 -> OCR识别 -> 在线编辑 -> 导出文档”的合同识别工具，优先满足中小体量合同处理场景，降低人工录入成本。

## 2. 业务范围
- 页面范围：2个页面（上传页、识别编辑页）。
- 用户分层：普通用户（最多10张）、高级会员（不限制）。
- 导出类型：`docx`、`pdf`。
- 本期不做：完整会员支付系统、多语言识别、复杂模板管理。

## 3. 关键用户故事
- 作为普通用户，我希望一次上传最多10张合同图并批量识别。
- 作为高级会员，我希望上传数量不受限制。
- 作为编辑者，我希望逐张对照原图修正识别文本。
- 作为使用者，我希望导出 `docx/pdf` 方便归档和发送。

## 4. 交互与功能规格

## 4.1 上传页（会话式 UI）
- 上传方式：点击上传、拖拽上传、多选上传。
- 预览形式：对话消息流中展示图片缩略图卡片，可点击大图预览。
- 限制校验：
  - 普通用户上传总数 > 10：拦截并提示。
  - 非支持格式/超尺寸：单项报错，不阻塞其他合法文件。
- 操作：删除单图、清空列表、开始识别。
- 状态：空态、已上传、识别中、识别失败。

## 4.2 识别编辑页（左图右文）
- 页面纵向排列识别卡片，每卡片：
  - 左侧：原图预览（缩放）。
  - 右侧：OCR文本编辑区（实时可编辑）。
- 卡片状态机：`idle` -> `processing` -> `success | error`。
- 单张失败支持重试；整体支持继续导出（失败页提示为空）。
- 右下角固定操作区：导出 `docx`、导出 `pdf`。

## 5. 导出规格
- 生成内容顺序：严格按上传顺序。
- 导出结构：
  - 文档标题：`合同识别结果_YYYYMMDD_HHmm`。
  - 每页前缀：`第N页`。
- 空文本策略：允许导出，导出前提示空页编号。

## 6. 技术方案（前端优先）
结论：前端可做 MVP，生产建议前后端协作。

### 6.1 MVP（纯前端）
- OCR：`tesseract.js`（浏览器端）。
- 状态：Zustand 管理上传项、识别状态、编辑文本。
- 导出：
  - `docx`：`docx` npm 包；
  - `pdf`：`jspdf` + `html2canvas`。
- 会员能力：先通过 `isPro` 配置/Mock 决策上传限制。

### 6.2 生产（推荐）
- 前端负责交互与编辑，后端负责 OCR 和会员校验。
- API预留：
  - `GET /api/user/plan`
  - `POST /api/ocr/batch`
  - `POST /api/export/docx`（可选）
  - `POST /api/export/pdf`（可选）

## 7. 非功能要求
- 性能：10张图内识别与编辑不卡顿，主线程可交互。
- 稳定性：核心流程成功率 > 95%（常规网络/图片质量）。
- 可观测：记录识别耗时、失败率、导出成功率。

## 8. 验收标准（DoD）
- 普通用户第11张上传被拦截，高级会员不拦截。
- 每张图片可预览且文本可编辑。
- 能导出 `docx` 和 `pdf`，内容顺序正确。
- 失败页可重试，不阻断其他页导出。
