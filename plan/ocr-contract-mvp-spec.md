# 合同识别与导出 MVP 可执行 Spec

## 1. 文档目标
基于当前“上传合同图片 -> OCR识别 -> 编辑 -> 导出”的业务，产出一份可直接落地执行的产品与技术规范，供开发 Agent 按步骤实现。

## 2. 范围与边界
- 本期仅做 2 个页面（上传页、识别编辑页）。
- 图片上传限制：普通用户最多 10 张；高级会员不限制（会员能力本期先预留接口，不实现完整会员系统）。
- 导出格式：`docx`、`pdf`。
- 本期先聚焦中文合同识别场景，支持手动纠错后导出。

## 3. 用户流程（E2E）
1. 用户进入上传页，选择/拖拽图片（可多选）。
2. 系统校验数量限制与格式合法性。
3. 用户点击“开始识别”，进入识别编辑页。
4. 每张图片显示“左图右文”编辑区：左侧预览原图，右侧显示 OCR 文本并可实时编辑。
5. 用户编辑完成后，点击右下角“导出”，选择 `docx` 或 `pdf`。
6. 系统按图片顺序拼接文本并导出文件。

## 4. 页面规格

## 4.1 页面1：上传与会话式交互页
### 功能点
- 会话式 UI（类似 Agent 对话区）：
  - 聊天主区显示系统提示与操作反馈（如“已上传 3/10 张”、“正在识别第 2 张”）。
  - 输入区支持上传按钮、拖拽上传。
  - 已上传图片以缩略图气泡形式展示，可点击放大预览。
- 数量限制：
  - `isPro=false` 时最多 10 张；
  - `isPro=true` 时不限制（前端逻辑 + 后端二次校验）。
- 文件校验：
  - 支持 `jpg/jpeg/png/webp`；
  - 单文件大小建议上限 15MB（可配置）。
- 操作能力：
  - 删除单张；
  - 清空全部；
  - 按上传顺序排序（默认按添加时间）。

### 交互状态
- 空状态：引导上传。
- 已上传未识别：显示图片列表 + “开始识别”按钮。
- 超限状态：toast + 阻止继续添加。
- 识别中：按钮 loading，禁止重复提交。

## 4.2 页面2：识别编辑与导出页
### 布局
- 垂直从上到下排列多个“识别卡片”。
- 每个卡片为左右结构：
  - 左侧：图片预览（支持缩放、上一张/下一张）。
  - 右侧：可编辑文本区域（Textarea/富文本均可，MVP建议纯文本）。

### 功能点
- 实时编辑：输入即保存到本地状态（可加 500ms 防抖持久化）。
- 卡片级状态：
  - `待识别` / `识别中` / `成功` / `失败`。
- 重试机制：
  - 单张识别失败可重试。
- 导出能力（页面右下角固定区）：
  - 导出 `docx`；
  - 导出 `pdf`；
  - 导出顺序严格按图片顺序。

### 导出规则（MVP）
- 标题：`合同识别结果_YYYYMMDD_HHmm`。
- 每张图片文本前增加分隔标题：`第N页`。
- 空文本页面允许导出，但在导出前提示“第N页为空”。

## 6. 技术方案与可行性分析
## 6.1 这个项目可以只用前端吗？
结论：**可以做出可用 MVP，但不建议长期只前端。**

### 仅前端可行（适合快速上线/内测）
- OCR：浏览器端 OCR（WASM/JS OCR 引擎）；
- 导出：
  - `docx`: 前端库生成并下载；
  - `pdf`: 前端排版后导出；
- 优点：开发快、部署简单、无服务端成本。
- 风险：
  - 识别质量受浏览器性能影响大；
  - 大图或多图时内存压力大；
  - 无法做统一模型升级与质量控制；
  - 会员限制只在前端不安全，容易被绕过。

### 推荐方案（生产）
- 前端 + 轻量后端（BFF/API）：
  - 前端负责上传、编辑、导出交互；
  - 后端负责 OCR 调用、会员校验、任务编排、日志审计。
- 收益：稳定性、可观测性、可运营性更好。

### RapidOCR 在项目中的定位
- `RapidOCR` 是 OCR 引擎，不是“完整后端架构”。
- 你看到的 Python 脚本属于本地离线识别示例，用于快速验证识别效果。
- 真正上线时建议把 RapidOCR 作为后端 `ocr-service` 的一个引擎实现，通过 API 给前端调用。

## 6.2 本期推荐技术栈（前端优先）
- 前端框架：React + TypeScript（沿用当前项目）。
- 状态管理：Zustand/Redux（二选一，MVP 用 Zustand 更轻）。
- UI：现有组件库 + 自定义会话式消息区。
- 导出：
  - `docx`: `docx`（npm）；
  - `pdf`: `jspdf` + `html2canvas`（或 `pdf-lib`）。
- OCR：
  - 纯前端：`tesseract.js`（MVP）；
  - 若接服务端：后端 `RapidOCR (Python)` + 前端封装 `POST /ocr/jobs`。

## 6.3 前后端架构设计（便于后续扩展）
- 前端（Web）：
  - 负责上传交互、识别结果编辑、导出触发、状态展示；
  - 不负责会员权限最终校验。
- API/BFF：
  - 负责鉴权、配额校验、路由编排、统一错误处理。
- OCR服务（Python）：
  - 使用 RapidOCR 做识别；
  - 负责预处理、结果标准化、重试。
- 导出服务：
  - 统一生成 docx/pdf（可先前端导出，后续切后端）。
- 基础设施：
  - 对象存储（图片、导出文件）；
  - 数据库（任务状态、识别结果、导出记录）；
  - Redis/队列（异步任务）。

## 7. API 约定（前后端分层版）
- `GET /api/user/plan` -> `{ isPro: boolean }`
- `POST /api/upload/images` -> 上传返回 `fileIds`
- `POST /api/ocr/jobs` -> 创建识别任务，返回 `jobId`
- `GET /api/ocr/jobs/:jobId` -> 轮询任务状态与结果
- `POST /api/export/jobs` -> 创建导出任务，返回 `jobId`
- `GET /api/export/jobs/:jobId` -> 返回导出状态与下载链接

> 说明：若本期纯前端，可先用本地 Mock 实现同样的数据结构，后续无缝切换。

## 8. 执行计划（Agent 可直接照此实现）
## Phase 1：上传页（1-2天）
- [ ] 实现会话式上传 UI 与图片预览气泡。
- [ ] 实现 10 张限制 + `isPro` 放行逻辑。
- [ ] 实现文件类型/大小校验与错误提示。
- [ ] 实现“开始识别”并跳转编辑页，传递图片状态。

## Phase 2：识别编辑页（2-3天）
- [ ] 实现“左图右文”卡片列表（按顺序）。
- [ ] 接入 OCR（先 mock，后接真实）。
- [ ] 实现文本实时编辑与状态保存。
- [ ] 实现单张失败重试。

## Phase 3：导出与联调（1-2天）
- [ ] 实现 `docx` 导出（含第N页分隔）。
- [ ] 实现 `pdf` 导出。
- [ ] 增加导出前校验（空页提示）。
- [ ] 完成基础埋点（上传数、识别耗时、导出成功率）。

## 9. 验收标准（DoD）
- 普通用户上传第 11 张时明确拦截；高级用户不拦截。
- 识别编辑页每张图片均可查看原图并编辑文本。
- 导出 `docx/pdf` 成功率 > 95%（10 张以内，常规图片）。
- 导出内容顺序与上传顺序完全一致。
- 页面无阻断级错误，核心流程可在主流 Chromium 浏览器跑通。
